# AUR packaging

These directories are the upstream copies of the three SayAll AUR packages:

- `sayall`: installs the official x86-64 GitHub Release binaries;
- `sayall-src`: builds the latest stable tag from source;
- `sayall-git`: builds the current `main` branch from source.

Each directory is published as its own AUR Git repository. Keep its `PKGBUILD`,
`.SRCINFO`, and `.install` file synchronized. The packages provide the same
commands and systemd user units, and therefore conflict with one another.

For a stable release update, change `pkgver`, reset `pkgrel` to 1, update
checksums, test with `makepkg`, regenerate `.SRCINFO`, and then push the package
repository to the AUR. VCS package versions are generated by `pkgver()` and do
not need manual updates for every commit.

`sayall` has `replaces=('sayall-bin')` for the one-time binary-package rename.
The source variants provide `sayall`, and all three current variants conflict
with one another and with the retired `sayall-bin` name. Each variant stores
licenses below `/usr/share/licenses/<pkgname>` while sharing the runtime paths
and `/usr/share/doc/sayall` documentation path.

## Public AUR transition

The existing `sayall` AUR repository is updated in place from its old source
recipe to the official prebuilt recipe. `sayall-src` is a new AUR repository;
`sayall-git` continues in place. The release workflow deliberately targets only
`sayall-src`, `sayall`, and `sayall-git`, in that publication order. It never
pushes the retired `sayall-bin` repository.

Before the first transitioned release, verify that the account named by the
`AUR_MAINTAINER` Actions variable maintains the public `sayall` and
`sayall-git` packages. `sayall-src` must either be unclaimed or maintained by
that same account. The dedicated publishing key must belong to that account.
The workflow performs this public ownership preflight and clones every target
before making its first change.

After all three current repositories have published successfully, request an
AUR merge of `sayall-bin` into `sayall` to preserve votes and comments. This is
a manual AUR action and must not happen before the new `sayall` recipe is live.
Existing `sayall-bin` installations still require the explicit migration in
the main README; an AUR merge does not rename an installed package.
